# Спека: домен payments (платная бронь и предзаказ)

**Дата:** 2026-07-21
**Статус:** черновик, три открытых вопроса по деньгам (§9)
**Ветка:** `feature_payments` (от `develop`)
**Зависимости:** волна 3 (bookings) — PR #4

---

## 1. Цель и рамки

Дать возможность брать деньги за бронь и за предзаказ блюд. Деньги принимаются на
BookEat, дальше идёт расчёт с заведением. Провайдеров два — **FreedomPay** и
**TipTopPay**, и их должно быть легко заменить или дополнить третьим.

Вне рамок этой волны: автоматические выплаты заведениям (реестр к выплате
формируем, само перечисление — вручную/бухгалтерией), лояльность, сплит-счета
между гостями.

---

## 2. Главный принцип: провайдер — сменная деталь

Домен НЕ знает ни про FreedomPay, ни про TipTopPay. Он знает про порт:

```go
// internal/domain/payment.go
type PaymentGateway interface {
    Authorize(ctx, AuthorizeRequest) (*GatewayPayment, error) // холд (двухстадийный)
    Capture(ctx, providerID string, amount Money) (*GatewayPayment, error)
    Void(ctx, providerID string) error                        // снять холд
    Refund(ctx, providerID string, amount Money) (*GatewayRefund, error)
    Get(ctx, providerID string) (*GatewayPayment, error)      // сверка
    VerifyWebhook(raw []byte, headers map[string]string) (*WebhookEvent, error)
    Name() PaymentProvider
}
```

Применяемые шаблоны и зачем именно они:

| Шаблон | Где | Что даёт |
|---|---|---|
| **Ports & Adapters** | `PaymentGateway` в domain, адаптеры в `infrastructure/payment/{freedompay,tiptoppay}` | новый эквайер = новый файл, домен не трогаем |
| **Registry / Strategy** | `map[PaymentProvider]PaymentGateway`, выбор по настройке заведения | переключение провайдера — настройка, не релиз |
| **Anti-corruption layer** | адаптер переводит коды и статусы провайдера в доменные | коды FreedomPay не растекаются по коду |
| **Transactional outbox** | `payment_outbox` | уведомления и вебхуки наружу переживают откат |
| **Idempotency key** | и на входе от клиента, и наружу к провайдеру | двойное списание невозможно |
| **State machine** | статусы платежа с таблицей переходов | нет «оплачено → снова ожидает» |
| **Ledger (двойная запись)** | `payment_ledger_entries` | деньги сходятся, есть чем ответить на спор |
| **Saga / компенсация** | бронь не создалась → холд снимается | нет денег без брони |

Двухстадийная схема (холд → списание) обязательна: она есть у обоих провайдеров
(TipTopPay `auth` + `confirm` + `void`) и это единственный честный способ
обработать «заведение не подтвердило бронь» — деньги гостя не списываются, холд
снимается.

## 3. Что списываем: состав суммы

```
Итого к оплате = сумма предзаказа (или депозит за бронь)
               + сервисный сбор BookEat (настраиваемый %)
```

Сервисный сбор — **отдельная сущность с настраиваемыми размером и названием**,
показывается гостю ДО оплаты отдельной строкой. Он привязан к услуге BookEat
(онлайн-бронь, предзаказ, гарантия столика), а НЕ к способу оплаты.

> Юридическая оговорка, не проверено: перекладывание эквайринговой комиссии на
> покупателя отдельной строкой ограничивается правилами платёжных систем.
> Формулировки «комиссия за оплату картой» / «эквайринг 3,5%» использовать
> нельзя до письменного подтверждения от FreedomPay/TipTopPay и юриста.

Все суммы — целые в тиынах (`bigint`), валюта хранится явно (`KZT`). Никаких
чисел с плавающей точкой в деньгах ни на одном слое.

## 4. Схема (миграция `0007_payments.sql`)

**payments** — намерение заплатить, одна строка на попытку оплаты:
```
id, booking_id, restaurant_id, user_id
provider           varchar        -- freedompay | tiptoppay
provider_payment_id varchar       -- id на стороне эквайера
purpose            varchar        -- deposit | preorder
status             varchar        -- created|authorized|captured|voided|refunded|partially_refunded|failed|expired
amount_minor       bigint         -- всего к оплате
base_amount_minor  bigint         -- депозит/предзаказ без сбора
fee_minor          bigint         -- сервисный сбор
currency           varchar(3)
idempotency_key    varchar        -- ключ, отданный провайдеру
authorized_at, captured_at, voided_at, failed_at, expires_at
failure_code, failure_message
created_at, updated_at
UNIQUE (provider, provider_payment_id)
UNIQUE (booking_id) WHERE status IN ('authorized','captured')  -- одна живая оплата на бронь
```

**payment_refunds** — возвраты (возможны частичные, поэтому отдельная таблица):
`id, payment_id, amount_minor, reason, provider_refund_id, status, created_at`.

**payment_events** — сырые вебхуки как есть: `id, provider, provider_event_id,
payload jsonb, signature_valid, processed_at`. Уникальность по
`(provider, provider_event_id)` — вебхук приходит повторно, обработать его надо
один раз.

**payment_ledger_entries** — двойная запись: `id, payment_id, account
(guest|restaurant|platform|acquirer), direction (debit|credit), amount_minor,
created_at`. Сумма по всем записям одного платежа обязана быть нулём — проверяется
тестом.

**payment_outbox** — как в bookings: событие пишется в одной транзакции с
изменением платежа.

**restaurants** (расширение, всё nullable = «берём из env»):
`payments_enabled bool`, `deposit_required bool`, `deposit_amount_minor bigint`,
`preorder_payment_required bool`, `service_fee_bps int` (базисные пункты, 350 =
3,5%), `payment_provider varchar`.

Базисные пункты, а не проценты дробью: 3,5% в float — это ошибка округления в
чужом кошельке.

## 5. Машина состояний платежа

```
created ──authorize──▶ authorized ──capture──▶ captured
   │                       │                      │
   │                       └──void──▶ voided      ├──refund(part)──▶ partially_refunded
   └──fail──▶ failed                              └──refund(full)──▶ refunded
authorized ──(истёк срок холда)──▶ expired
```

Правила:
- Терминальные: `captured` (до возвратов), `voided`, `failed`, `expired`,
  `refunded`.
- Любой переход: `UPDATE payments` + запись в ledger + событие в `payment_outbox`
  — одна транзакция, как в bookings.
- Источник правды о статусе — вебхук провайдера, но никогда не только он:
  фоновая сверка (`reconcile`) раз в N минут дергает `Get` по всем висящим
  платежам. Вебхук может не дойти; деньги от этого не должны потеряться.

## 6. Сценарии

**Бронь с депозитом**
1. Гость создаёт бронь → она `pending`, платёж `created`.
2. Гость платит → холд (`authorized`). Бронь подтверждается по обычным правилам.
3. Заведение подтвердило → списание (`captured`) в момент подтверждения.
4. Заведение отклонило / бронь не подтверждена в SLA → `void`, холд снят, гость
   ничего не потерял.
5. Отмена гостем до дедлайна → возврат (по §9.2, ждём решения).
6. Отмена после дедлайна → возврата нет.
7. Неявка → деньги остаются заведению (в реестре к выплате).

**Предзаказ** — всегда платный: холд при оформлении, списание при подтверждении
брони. Отмена предзаказа заведением (нет блюда) → частичный возврат на позицию.

**Компенсация:** если бронь не создалась (проиграна гонка за стол), холд
снимается тем же запросом — иначе у гостя висят деньги за бронь, которой нет.
Это единственное место, где допускается вызов провайдера внутри обработки
запроса; он идемпотентен по ключу.

## 7. Вебхуки

- Отдельный публичный маршрут на провайдера:
  `POST /webhooks/payments/freedompay`, `POST /webhooks/payments/tiptoppay`.
- Первым делом — проверка подписи. Не прошла → 401 и запись в `payment_events`
  с `signature_valid=false`. Никакой обработки до проверки.
- Идемпотентность по `(provider, provider_event_id)`.
- Отвечаем 200 быстро, обработка — из очереди/воркера. Провайдер не должен ждать
  нашу бизнес-логику.
- Вебхук никогда не создаёт платёж «из воздуха»: событие на неизвестный
  `provider_payment_id` логируется и уходит в разбор, а не создаёт запись.

## 8. Безопасность и деньги

- Ключи провайдеров — только в env, никогда в базе и в репозитории.
- Суммы к оплате считает СЕРВЕР. Приходящая с клиента сумма игнорируется —
  иначе гость оплатит бронь на 1 тиын.
- Все обращения к провайдеру идут с нашим ключом идемпотентности; повтор при
  таймауте не создаёт второй платёж.
- Логи: никаких номеров карт, CVV и криптограмм. Только id платежа и статус.
- Возврат больше остатка невозможен — проверяется до вызова провайдера и
  ограничением в БД.

## 9. Открытые вопросы к владельцу

1. **Два провайдера сразу или один основной + запасной?** От этого зависит,
   нужен ли выбор провайдера на уровне заведения (заложено как настройка) или
   достаточно глобального.
2. **Возврат до дедлайна:** гость получает всю сумму, а наш сбор мы теряем? Или
   возвращаем за вычетом сбора? (Формулировка «удерживаем комиссию, и её же
   платим» допускает оба прочтения.)
3. **Неявка:** заведению уходит вся сумма целиком, включая наш сервисный сбор,
   или за вычетом него?

Пока ответов нет, расчёт возвратов вынесен в одну функцию `refundPolicy(...)` с
явными вариантами — переключается настройкой, переписывать домен не придётся.
